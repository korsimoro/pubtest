DPKI In Three Parts (ROUGH DRAFT)

This paper presents the design of a decentralized public key infrastructure, or a DPKI.  In this paper, we describe a protocol for discovering, distributing, and revoking key pairs by binding them to immutable identifiers in a non-repudiable manner.  The main improvement over existing work is that our system does not rely on any centralized certificate authorities, but instead relies on a handful of guarantees that are provided by blockchains today.

In describing this system, we motivate the need for decentralization as a first-class design principal, and we describe how to efficiently discover and contact the correct blockchain in the presence of computationally-bound adversaries.  We show how our system leverages properties of a blockchain’s construction to defeat an adversary who cannot completely block access to the network.  We then present a protocol for automatically managing a user’s private keys by mapping them onto existing security-related user experiences (e.g. 2FA), and describe how to use a blockchain to solve the private key loss and private key compromise problems.

It is separated into three parts:

  • Registration and Discovery (of identifiers)
  • Security (of identifiers)
  • Recovery (of lost or compromised identifiers)

Identifiers / Attributes / Registration / Discovery

Section Contributors Alphabetical By Last Name: Jude Nelson, Drummond Reed, Christian Lundkvist, Markus Sabadello, Greg Slepak

Previous public key infrastructures (PKIs), such as X.509, have attempted to ensure that an identity (usually associated with a human readable identifier, such as a domain name), does in fact belong to the owner of that identifier. The problem, however, is that such systems delegate the real ownership of that identifier to trusted third parties called Certificate Authorities (CAs), each of whom is able to sign statements claiming ownership of the identifier.

This behavior is problematic. It means that end-users cannot be sure that they are in fact getting the public key that they expect. For example, if they ask for the public key of twitter.com, many entities who are not Twitter are able to send them their own keys and convincingly deceive the user into believing the key belongs to Twitter. [Insert ref to estimate of number of CAs able to make such a statement, on the order of 1200+]

DPKI, or decentralized public key infrastructure, attempts to solve not just this problem, but also the problem of private key loss and recovery.

Registration

DPKI uses secure decentralized key-value datastores to allow anyone to register arbitrary keys (human-readable or unreadable identifiers) in a namespace according to the rules of that namespace. The rule can be as simple as first-come-first-serve, or as complicated as an auction system, or any other imaginable way of servicing registration requests. The registration process may consist of multiple steps, e.g. announcing an intent to register, potentially an opportunity for disputing such intents, and finally the actual registration step.

The registration process must be neutral and available to all members of the public and should not discriminate against any party that wishes to use it.

Once a user has successfully registered an identifier in a DPKI, they are able to prove ownership of that identifier because it is associated with a private/public key pair. So long as they do not let the identifier expire, and are able to maintain control of their private key, no other third-party can take ownership of that identifier from them without resorting to direct coercion of the owner.

Users may not interact with the datastore directly but instead use the services of an intermediary party (a registrar), who facilitates the registration and identifier management processes. Of course such an intermediary party needs to be trusted by the user to control their private key, which can obviously be problematic as examples in the Bitcoin community have shown [todo add links]. If such an intermediary party is used, there should be mechanisms that allow the user to arbitrarily switch to the services of another intermediary. Of course users should also be able to download their private keys at any time and start managing their identifier themselves.

Discovery

One form of decentralized datastore of suitability to DPKI that provide these properties are blockchains.

Blockchains, such as Namecoin, Bitcoin (via tech like Blockstore), Ethereum, and other blockchains, are capable of providing key-value lookups in a namespace (or namespaces). These namespaces can be used to for the purpose of website domain registration, or registration of individual identities.

By decentralizing consensus and control of the creation of a distributed ledger, blockchains make DPKI possible and ensure its properties, so long as the decentralization and security properties of the blockchain are sufficient.

Once a user registers an identifier on the blockchain, they can associate any data (a value) for that identifier. Blockchains are able to return that value in response to a request, and the data returned can include arbitrary information, including public keys and attributes (like IP addresses) associated with that identifier.

Most users will not be able to run full blockchain nodes, however, and therefore secure discovery must be done using thin clients, or if thin clients are not available, a unanimous quorum of trusted nodes. This mechanism is covered in detail in the Security section of the paper.

Identifiers and User Experience

One of the challenges with PKI technologies such as PGP is the process and user experience for establishing relationships and exchanging keys, which enable secure communication and data sharing. Unlike PGP which only exposes the “secure” and “decentralized” properties of Zooko’s triangle, for DPKI we envision support for understandable (semantic) identifiers to be associated with keys and potentially other information such as network service endpoints, in order to make secure digital identity and communication systems more accessible and easier to use.

A DPKI system can potentially be based on a variety of different identifiers, which have different properties with regard to security, privacy, memorability, ease-of-use, etc.

Our recommendation is that DPKI, a primary identifier should not be a public key, because the corresponding private key can get compromised or lost. We should also avoid identifiers that are locked to any specific underlying transport protocol. A longer term solution is a UUID or other strongly unique identifier that can be registered along with an associated signing key, or public key, or links to authorities for keys, revocation or identity. This immutable identifier would be used for querying identity and other public keys. For an introduction to objectives and requirements around such immutable identifiers, see Global Permanent Identifiers.

Human friendly identifiers can rest in layers on top of these unique identifiers.  These human-friendly identifiers may globally unique, or alternatively they may be discoverable (and verifiable) within a given context.  This latter approach is also known as "linked local names" or “petnames”.

To frame the discovery problem, suppose Alice wants to discover Bob’s public key.  To do so, Alice must know the immutable identifier Bob used to establish his public key with DPKI.  The immutable identifier is arbitrary, but must be globally unique. To make his identifier discoverable, Bob announces semantic names that map to it which Alice can find and verify (Figure 1).  Alice will be able to confirm their authenticity once she discovers Bob’s public key.

Figure 1: The relationship between Alice, Bob’s semantic identifiers, and Bob’s immutable identifier.  In this figure, Twitter, Google+, and Yahoo! resolve semantic identifiers to Bob’s immutable identifier in order to help Alice find it.  However, Alice does not trust these resolvers, because once she discovers Bob’s public key, she uses it to verify a document Bob wrote to confirm that he placed them there.

Threat Model

Section Contributors Alphabetical By Last Name: Jude Nelson

Like conventional PKI systems, DPKI assumes that a persistent active adversary constantly tries to trick Alice into receiving the wrong key for Bob.  This can take the form of discovering the wrong identifier, or the wrong key once the identifier is known.  The motivation for DPKI stems from recent revelations that centralized PKI services (such as certificate authorities) have been and continue to be compromised or compelled to issue invalid public keys.  This outcome is particularly devastating when the compromised CA is trusted by many users.

We assume that the adversary has the power to compromise individual CAs and cause them to issue invalid keys, and we assume that the adversary can temporarily disrupt communication between users, for at most T seconds.  However, we assume that the set of DPKI users combined controls more compute power than the adversary.  Real-world examples of such adversaries include organized criminal organizations, rogue ISPs, and botnets.

Our solution is to make use of a widely-used blockchain that exceeds the adversary’s computational power.  A blockchain is a peer-to-peer distributed append-only log of transactions with the following three properties:

  • Permissionless:  Blockchain peers accept any transaction, as long as it is well-formed.  No authentication is necessary to read or write transactions.
  • Inimitability through Proof-of-Work:  The blockchain peers accept a new block not only if it is well-formed, but also if it is accompanied by a solution to computational puzzle that is known to take a long time to solve (a “proof-of-work”).  Moreover, peers resolve conflicts between two diverging blockchains by choosing the one with the most cumulative proof-of-work, and peers work together to enforce that the proof-of-work required ensures a fixed average time between blocks.  This is to ensure that an adversary must invest at least as much computation (in a shorter amount of time) to create an acceptable blockchain forgery, which is a very costly proposition by design.
  • Ancillary Transaction Data:  The transaction format must allow for a (small) amount of sender-chosen data to accompany it.

We use a blockchain in DPKI to broadcast the sequence of bindings and unbindings between immutable identifiers and public key hashes.  We show that as long as (1) Alice and Bob can identify and receive data from the longest fork of the blockchain, and (2) the adversary does not exceed 50% of the compute power of the users producing it, then the adversary will be unable to trick Alice into discovering the wrong key for Bob.

Public Keys / Security

Section Contributors Alphabetical By Last Name: Vitalik Buterin, Jude Nelson, Greg Slepak

The degree of decentralization plays a role in security. Centralized systems are vulnerable to manipulation, censorship and attack. They require that users trust that the entity does not misbehave. If a centralized system goes down, it takes all its users with them.

Goal

The goal of a DPKI is to give end-users as much control of their online identity as possible. In X.509, control of identities rests not with end-users, but with trusted third parties called Certificate Authorities.


The more decentralized a PKI is, the more likely it will be able to achieve to provide end-user with real control of their human-readable identifiers.

Human Readable Identifiers As Mappings To Keys

In DNS and X.509, a human-readable identifier is mapped by DNS to an IP. That IP sends a certificate. The certificate is declared valid if it contains the identifier that was requested and if the certificate is signed by any CA that the client software trusts.

In such a system, identifiers are not effectively owned by their users, but by these third-parties. CAs are capable of creating arbitrary certificates for arbitrary identifiers.

The DPKI, however, acts as a secure key-value store where keys are the identifiers, and the values for those keys can only be specified by the entity that originally claimed ownership of that key (up until the moment of key-expiration).

In such a system, the worst that can happen is:

  • An outdated value for a key is sent in response to a lookup.
  • The owner of the key is not able to update the value for their key due to censorship, and they lose ownership once the key expires.

These problems are addressable through the use of thin clients, onion routing, etc.

It is also possible, although extremely unlikely, that a false value is sent for a key. This can happen, for example, if a blockchain secured by proof-of-work has an adversary who was able to overpower the honest nodes and reverse history beyond the point of registration. However, all participants of the system would be able to detect this attack because it would result in the orphaning of an extremely long chain.

Decentralization Metric

The decentralization of a system can be roughly measured as:

How many doors to knock on to compromise the users of a system?

Technology for building a DPKI

New decentralized datastores, like blockchains, can provide end-users more control over their identities. We can generally refer to blockchains as being an example for a technology of a decentralized key-value datastore. For convenience, we will refer to this class of technologies simply as decentralized datastores.

It’s important to note that decentralized datastores can become centralized, and so it’s important to have a metric that users can use to determine whether or not a “decentralized datastore” is actually decentralized.

Measuring The Decentralization Of Blockchains

We note that the (de)centralization of a blockchain depends on the following components:

  • “Devs” — The number of controlling parties who have control over the behavior of the blockchain (its source code).
  • “Nodes” — The number of blockchain replicas, measured by the number of full nodes.
  • “Validators” — The number of blockchain miners/validator, who are responsible for creating new blocks and authorizing transactions.

We define the decentralization of a blockchain as:

        Decentralization(Blockchain) = MIN(“Devs”, “Nodes”, “Validators”)

Another informal metric is simply the behavior of the datastore itself. Users can infer the decentralization of a datastore by the Quality of Service (QoS) that the datastore provides. If, for example, they notice that they are suddenly unable to update their identifiers (possibly due to censorship), then for them that datastore is not serving its purpose.

A Note On Access To Decentralized Datastores

Another, separate concern, is whether these decentralized key-value datastores are accessible to end-users.  For example, a Bitcoin is not useful to users who cannot access it due to censorship at the ISP level. We note, however, that this is also true for users who don’t have access to computers.

Access to decentralized datastores is a separate, orthogonal topic to the mere existence of a DPKI (once accessed), and is separately addressable by other technologies such as mesh networking, onion routing, etc.  It is nevertheless an important consideration, because we expect that users will rely on external untrusted data stores to host signed information.

In particular, once Alice discovers Bob’s public key hash, she needs to obtain the public key.  The hash allows her to reject invalid data, which means that Bob’s choice of storage provider is arbitrary.  It could be a commodity cloud storage provider, Bob’s personal website, a legacy key server, or a DHT, and so on.  This is critical to thwarting the adversary, since it increases the availability of the key beyond what a single provider can offer.

Once Alice obtains Bob’s public key, she can use it to verify the authenticity of any subsequently-fetched data that identifies Bob as the author (such as a list of semantic names).  Similarly, this data can be replicated to any host of Bob’s choosing.

The DPKI Specifies A Datastore Agnostic Protocol

If a DPKI were to specify a specific blockchain as its “defacto decentralized datastore”, that in itself would put centralization pressures on that blockchain.

For example, if Ethereum were named the defacto datastore for a DPKI, it would be problematic for end-users should the blockchain become centralized because software developers, having coded support for a specific blockchain, would have to expend more effort to rewrite that software to support a different blockchain. Therefore, the use of an agnostic protocol for accessing decentralized datastores is a requirement to ensure the decentralization of the DPKI as a whole.

Agnostic protocols make it easier for users and developers to migrate to a different datastore that better serves their needs, and the mere existence of this possibility increases the quality-of-service (QoS) that every participating datastore provides through the creation of a market of decentralized datastores.

Security

One of the challenges with using blockchains for any kind of authentication protocol that seeks to establish mainstream usage is: how do clients access the blockchain? The simplest solution is for users to run a “full node”, downloading all blocks and all transactions and themselves maintaining the state of every blockchain application. However, particularly for low-power devices this is problematic; the amount of data stored increases with the product of usage and time, and so in practice, even with advanced pruning techniques, the amount that needs to be downloaded can exceed 30 GB and the amount that needs to be stored can exceed 1 GB, alongside other bandwidth and computational costs. Hence, it is not always reasonable to expect nodes to run full nodes; however, if they do not, how else can they “read” what the blockchain is saying?

One simple solution is to have a set of “blockchain notary” intermediaries that tell users what the state of a particular object maintained by a blockchain is. However, this route arguably compromises that was the key purpose of blockchain technology in the first place: removing the need for trusted intermediaries. Fortunately, there is another technological solution: thin-client protocols.

Thin-client protocols typically work in two stages. First, the thin client downloads only the header chain: a very small amount of information (typically 80-600 bytes) for each block containing metadata such a proof of work nonces as well as the root of a cryptographic hash tree (such as a Merkle tree) containing data such as transactions and possibly the state of the application that the blockchain keeps track of. The client then validates the header chain by using the blockchain’s underlying consensus algorithm (eg. checking proof of work or proof of stake signatures). After that point, the client treats the header chain as “trusted”, and uses cryptographic techniques that use the data in the header chain as a “root hash” from which it can verify claims about the rest of the data stored in the chain.

Fetching the Header Chain

The first challenge for a thin client is to download and verify the header chain. Assuming a working network connection, this is easy. For example, in the proof of work case the client asks the network for as many block headers as it can provide, the network replies back, and the client checks to make sure that each header has valid proof of work and then determines the “longest” chain of valid block headers (where “longest” is taken to mean “represents the most cumulative work”). More advanced protocols using skiplists exist by which clients do not even need to download every block header, though in-depth discussion of this is beyond the scope of this paper.

The main challenge with this mechanism is simple: what if the network connection is compromised? Potentially, an internet service provider could attack a user by censoring replies that tell a client about the official chain, and instead tell the user about their own fork. With proof of work protocols, one can statistically detect this by noticing a reduction in the rate of block production; however, there is much research left on determining the best and most reliable way to do this.

Merkle Trees

The second step in the 1-to-N step: given that you have successfully received a small piece of data that is “trusted”, how do you verify claims about the rest of the data in the chain? The core of the solution relies on Merkle trees. A Merkle tree is a hashing algorithm where a large number of “chunks” of data is hashed a few pieces at a time, and then the resulting hashes are themselves put into small groups and hashed, and so on recursively until the process results in one single hash, called the root. The benefit of this is that the membership of any single chunk of data in the tree can be proven via a Merkle branch: the subset of nodes in the tree whose values are used in the process of computing the root hash.

With just this set of nodes, a thin client can verify that a particular chunk is in the tree that has a particular proof, and the scheme is secure up to collision resistance: in order for an attacker to cheat the scheme, the attacker would need to break the underlying hash function. There are many different kinds of Merkle trees, including simple binary trees and more advanced designs such as Merkle Patricia trees that allow for efficient insert and delete operations, but the basic principle is the same.

What do we want to know?

Using header chains and Merkle trees as our primary tools, we can verify many kinds of claims about the data in the chain. But what kind of information do thin clients want to know?

The following are some examples:

  • Determining the time that a particular record was created or first seen
  • Finding the public key currently associated with an account ID
  • Finding the IP address and other data currently associated with a domain name
  • Learning about key revocations (with no specified process for finding a replacement key)
  • Determining the most recent version that a developer has published for a particular software package

In some cases, these problems can be decomposed into a “find the most recent transaction signed by a particular private key” problem. In other cases (eg. domain name registration), more complex logic is employed. One moderately complex pattern that might be commonly applied for account management is a key revocation setup where there exist multiple keys of which the majority can always revoke the minority but where the majority can naturally change over time.

More generally, we can decompose this into three categories of problems:

  • Proving existence: proving that an event of a particular kind happened at time T
  • Proving inexistence: proving that an event of a particular kind did not happen between times T_1 and T_2
  • Proving state: proving that the “state” of an application, which can potentially be a result of complex “state transition” rules from applying many transactions, is equal to X at time T

These are roughly arranged in order of hardness; proving existence is the easiest, and proving state is the hardest.

Degrees of Security

In general, we believe there are three levels of security that a thin-client protocol on top of a blockchain can offer:

  • Maximum security: if a thin client makes a query, and any node responds with a valid reply to that query, then (provided we can detect block withholding attacks) the thin client can immediately either learn (i) the correct answer to the query, or (ii) that the response was invalid and should be ignored.
  • 1-of-N trust security: if a thin client makes a query, and it is accepted as a security assumption that at least one honest node will respond correctly within T seconds, then the thin client can learn the correct answer after T seconds, regardless of how many offline/faulty/byzantine nodes there are.
  • N/2-of-N trust security: if a thin client makes a query, it must select some set of nodes that it trusts to respond, and it will learn the correct answer as long as the majority of the nodes are not offline or colluding to defraud it.

As an example of how these models apply, consider the simple case of “finding the current public key associated with an account”, assuming that there exists a single master key (or set of master keys) that has the right to revoke and replace keys. Suppose that we have a blockchain where only transactions are placed in Merkle trees. Then, a thin client sends a request asking the network for a Merkle proof of the most recent transaction that replaced the public key. If the client receives an answer, it knows:

  • With the maximum security assurance that this was the valid key at some point in the past
  • With 1-of-N trust security assurance that this is still the valid key (theoretically, a newer replacement transaction could exist, but a 100% collusion or censorship could lead to the client never learning about it)

For protocols that have more complex needs (eg. implementing complex name registrar rules), clients would only know the answer with N/2-of-N trust security assurance. However, if we have a blockchain where the state is in a Merkle tree, then the client can learn absolutely any fact with maximum security assurance.

Because different blockchains have different levels of usage of Merkle proofs, our proposed solution is to develop an abstract protocol by which different blockchains can be used (as no single blockchain is 100% guaranteed not to be fatally flawed, and so we want an abstract model similar to that used for encryption algorithm choices), and which automatically attempts to provide the best level of security available depending on the blockchain’s capabilities. Notaries would be available as a backstop, but blockchain plugins would exist which the client could install to support specific blockchains, and these plugins would intelligently make the kinds of blockchain queries that provide as much security as possible.

Private Key Management & Recovery

Section Contributors Alphabetical By Last Name: Vitalik Buterin, Christian Lundkvist,  Pavel Kravchenko, Jude Nelson, Duke Dorje, Arthur Brock, Greg Slepak, Noah Thorp, Harlan T Wood

One of the main barriers to entry in deploying DPKI is that it requires users to manage their private keys.  This represents a significant operational challenge in practice, in which we identify two main “pain points”:

  • Securely sharing a private key across the user’s devices
  • Recovering from either a key loss or compromise.

Our solution to these problems depends on the existence of a widely-used blockchain.  

We show that by using a blockchain to broadcast a sequence of public keys for an immutable identifier, DPKI can handle private key management without requiring the user to know how PKI works.

Private Key Distribution

Because a DPKI-reliant application requires the user to sign messages sent to it, the user’s device necessarily needs access to the user’s private key.  It is difficult to allow this securely, because the trust the user places in the device to keep the private key a secret is based on the context in which it is used.  For example, a user might trust their laptop to host their private key, until the laptop is lost or stolen.  As another example, a user might trust their mobile phone to host their private key, but not when the user shares it with a friend.

The contextual nature of trust implies that a general solution should minimize both the amount of information with which a device is trusted, as well as the amount of time for which it is trusted.  This is at odds with satisfying ease-of-use, because it also implies that a user should not simply copy the private key to each device she will use and leave it there.  To this end, DPKI must address the private key distribution problem in a secure, automatic manner, to minimize the operational costs to users without reducing security.

Private key distribution can be solved this way if we observe two constraints.  First, we assume that the user trusts the device while it is being used.  Second, we assume that the user has the ability to establish a throw-away secret across all devices.  We believe these are reasonable in practice, since these can be incorporated into familiar user experiences such as establishing 2FA or device pairing.

We can realize private key distribution via the following.  A user breaks the private key into multiple shards that are useless by themselves but such that M of N of them (M < N) can be recombined to form the original private key[b] (N is the number of devices to hold different shards).  Using the throw-away shared secret, the user encrypts and distributes both the public key and the private key shard to each device.  Then to use the private key, the user pulls the shards from M devices and reconstructs the private key (verifying that it is the correct private key by using it to regenerate the stored public key).

Critically, each of these steps is automated with client software.  The user experience for sharing the key shards reduces to “pairing” devices, such as via QR codes or PIN numbers.  The experience for recombining shards reduces to multi-factor authentication requests, whereby each shard-bearing device requests the user’s permission (or otherwise alters the user) that the shard is about to be shared.  The user is not exposed to the keys directly.

Private Key Revocation

Once a private key is established, there are three scenarios where a user would need to revoke it: the private key gets lost, the private key gets compromised, or the private key violates its usage policy (i.e. it expired, it was used too often, etc.).  We show that the latter two scenarios can be addressed automatically if a solution exists for the first scenario.

Private key loss is a devastating but common problem whereby the user no longer has the key with which to authenticate to other principals via PKI.  As such, any request to change the private key must be generated while the user still has the key, such as when the user creates it publishes it.  This information traditionally takes the form of a revocation certificate.

Managing revocation certificates imposes both operational and security challenges to the user. First, the user must either store and broadcast revocation certificates themselves (an operational problem), or must select trusted parties that will broadcast them on their behalf, but only with their say-so (a security problem).  Selecting these parties is difficult because it is hard to know whether or not they can be relied upon to publish the certificate in the future, and it is hard to ensure that the revocation certificate will be broadcast in a timely fashion.

Another critical limitation of revocation certificates is that they only disable the use of a key, not an identity.  Because PKI binds an immutable identifier to its current public key, an adversary can steal a user’s revocation certificate, revoke the user’s current key, and establish an attacker-controlled key as the current key for the user’s identity (which the user cannot revoke).  In other words, a secure key revocation protocol must prevent identity theft, in addition to key theft.

Our solution in DPKI is to ensure that an identity can only change its public key if the new public key is signed by a user-chosen M of N principals.  [c][d][e][f][g][h][i]This is to allow the user to choose their own trust model.  For example, one user might allow their three of her five closest friends to switch their public key, while another user might instead require three of five signatures from secret keys stored in five separate bank vaults, or possibly a combination of a friend, their employer and a specialized key storage company.  The only constraint is that each time the changes in this manner, the user identifies the next M of N key pairs which must sign the new key.

Importantly, this scheme does not require users to trust someone else to revoke a key on their behalf because it decouples the revocation mechanism from the revocation policy.  We believe this is an important property of the system, for two reasons.  First, it is sufficiently general to encode any complex key revocation policy as a composition of small, simple policies:  because the bearer of each of the M keys independently stipulates their own acceptance criteria for a reset signature, a set of M bearers defines a policy of M independent conjunctions that must hold true.  Second, it allows for market-based trust solutions to evolve and innovate a bearer’s acceptance criteria independently of the underlying technology.

We enforce this revocation protocol by binding the identifier/public-key pair to a multi-signature transaction in the blockchain.  This not only enforces the M-of-N signature scheme, but also ensures that 3rd parties running alongside blockchain peers observe the latest key and the latest set of N public keys in a bounded amount of time.
